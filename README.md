# 🚀 LLMOps CI/CD (Quality Gate) 시뮬레이션 실습 

이 저장소는 복잡한 인프라 설정이나 토큰 발급 과정 없이, **GitHub Actions가 어떻게 LLM의 배포 품질을 통제하는지(Quality Gate)** 그 원리를 직관적으로 이해하기 위한 교육용 미니 프로젝트입니다.

---

## 🎯 실습 목표

코드베이스의 변경사항(예: 프롬프트 업데이트)이 발생했을 때,
1. GitHub 서버가 자동으로 이를 감지하여 평가 스크립트를 실행합니다.
2. 평가 점수가 내가 설정한 **기준 선(Quality Gate, 예: 0.85)**을 넘지 못하면 배포를 시스템 레벨에서 **강제 차단(Block)**합니다.
3. 통과(Pass)할 경우에만 다음 워크플로 단계(예: 가상 배포)로 진행되는 것을 눈으로 확인합니다.

---

## 👨‍💻 실습 방법 (따라해 보세요!)

이 실습은 `evaluate.py` 파일 내의 `FAKE_SCORE` 점수를 조작하여 성공/실패 사례를 번갈아 관찰합니다.

### ❌ (시나리오 1) 품질 미달 코드 배포 차단 (Fail 테스트)

1. `evaluate.py` 파일을 열어봅니다.
2. 10번째 줄 부근의 `FAKE_SCORE = 0.70` (또는 0.85 미만의 값)로 설정되어 있는지 확인합니다.
3. 이 상태로 코드를 커밋(Commit)하고 GitHub 저장소로 Push 합니다.
4. GitHub 저장소 상단의 `Actions` 탭으로 이동합니다.
5. 방금 실행된 워크플로가 **빨간색 ❌ (실패)** 아이콘으로 끝나고, 배포 단계가 취소된 것을 확인합니다.
   * _"성능이 떨어지는 환각(Hallucination) 심한 결과물이 고객에게 나가는 것을 막아냈습니다!"_

### ✅ (시나리오 2) 품질 개선 및 배포 승인 (Pass 테스트)

1. 다시 `evaluate.py` 파일을 엽니다.
2. 이번에는 성능이 개선되었다고 가정하고 `FAKE_SCORE = 0.95` (0.85 이상의 값)로 수정합니다.
3. 수정한 내용을 저장하고 다시 커밋 & Push 합니다.
4. GitHub 저장소의 `Actions` 탭으로 이동합니다.
5. 이번에는 워크플로가 **초록색 ✅ (성공)** 아이콘으로 빛나며, 마지막 서버 배포 스텝까지 실행된 로그를 확인합니다.
   * _"훌륭한 품질이 검증되어, 코드가 안전하게 운영 서버에 배포되었습니다!"_

---

## 💡 이것이 왜 중요한가요? (핵심 개념)

LLM 애플리케이션 개발은 한 줄의 프롬프트 변경만으로도 모델의 대답 질감이나 정확성이 크게 요동칠 수 있습니다. 

개발자가 일일이 "이 질문해보고, 저 질문해보고..." 수동 테스트를 하는 것은 시간 낭비일뿐더러 사람의 실수를 유발합니다. 이렇게 GitHub Actions 같은 CI/CD 도구와 RAGAS 같은 평가 프레임워크를 연동해두면, **"객관적으로 채점하여 통과한 안전한 코드(프롬프트)만이 실서비스에 올리는"**는 튼튼한 방어벽이 구축됩니다.

이것이 바로 **LLMOps 평가 파이프라인 자동화**입니다.
